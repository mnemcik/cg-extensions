**Spec-driven development for greenfield software builds.** When a project is a **new (greenfield) software build** — code that does not yet exist — and the user wants a PRD and/or a layered specification (asks for "a spec", "PRD", "spec authoring", "ADRs", "spec-driven development"), load [`notes/spec-driven-development.md`](notes/spec-driven-development.md) for the four ground rules, the three phases (PRD → specs → milestone implementation), the spec-authoring step list with applicability flags, and the prompt templates. **Does NOT apply to** brownfield / existing-codebase work (changes, refactors, bug fixes, or features inside a codebase that already exists), nor to research, analysis, event, or documentation projects — even when one of those asks for "a spec". For those, treat the request literally and skip this workflow.
